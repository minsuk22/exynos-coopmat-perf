# MATMUL에서 FP16 / INT8 성능 차이가 사라지는 이유 (한글)

## 관찰된 현상

- 이 GPU의 연산 피크: **FP16 = 8 TOPS, INT8 = 16 TOPS** (INT8이 2배).
- **WMMA 단독 테스트**(`wmma_peak.comp`): 예상대로 성능이 나오고 **데이터 타입별 차이(INT8이 FP16의 2배)** 도 그대로 보임.
- **MATMUL 테스트**(1024×1024×1024 GEMM): FP16/INT8 **둘 다 ~3 TOPS로 동일**, 타입 차이가 사라짐.

## 한 줄 요약

**MATMUL이 메모리(데이터 공급)에 막혀 WMMA 연산기가 굶고 있기 때문입니다.** 연산기가 노는 동안에는 INT8의 2배 연산 우위가 발현될 수 없고, 둘 다 "데이터 공급 속도"라는 같은 천장에 걸려 같은 값으로 수렴합니다.

## 왜 WMMA 단독은 차이가 나는데 MATMUL은 안 나나

| | WMMA peak 테스트 | MATMUL (GEMM) |
|---|---|---|
| 데이터 | 16×16 타일이 레지스터/캐시 상주 | 1024² 행렬을 메모리에서 계속 로드 |
| 병목 | **연산기(ALU) 처리량** | **메모리 / 로드 공급** |
| 결과 | ALU 능력 그대로 → FP16 8, **INT8 16 (2×)** | ALU가 굶음 → 공급 속도에 캡 → **둘 다 ~3** |

- **WMMA peak = compute-bound** → 연산기 속도가 그대로 측정됨 → INT8이 FP16의 2배 (스펙 8 vs 16과 일치).
- **MATMUL = memory-bound** → 연산기가 로드를 기다리느라 대부분 idle → INT8의 2배 연산력은 쓸 일이 없어 안 보임.

## 왜 "INT8이 2배 빠른" 게 아니라 "똑같이" 나오나 (핵심)

만약 병목이 **바이트 대역폭**이라면, INT8 입력은 FP16의 절반 바이트이므로 **INT8이 ~2배 빨라야** 합니다. 그런데 **똑같다**는 것은 병목이 바이트가 아니라 **"개수"** 라는 뜻입니다. MATMUL에서 시간을 지배하는 요소가 전부 타입과 무관합니다:

1. **coopMatLoad 개수가 타입과 무관**: FP16이든 INT8이든 같은 M·N·K, 같은 타일링이면 로드하는 16×16 타일 **개수가 동일**합니다. 메모리/L2 → 레지스터 경로가 **원소·타일 개수 단위**로 율속되면 8비트든 16비트든 같은 속도 → 같은 OPS.
2. **WMMA 명령 개수도 타입과 무관**: `coopMatMulAdd` 발행 수가 동일합니다. 연산기가 굶어서 발행/대기로 시간이 정해지면 둘 다 같은 시간.
3. **출력/누산은 둘 다 4바이트**: FP16→**FP32**, INT8→**S32**. C 읽기 + D 쓰기 트래픽이 타입과 무관하게 동일합니다.

즉 **로드 타일 수, WMMA 명령 수, 출력 바이트가 전부 타입과 무관**하므로 FP16/INT8이 같은 ~3 TOPS로 수렴합니다. INT8의 유일한 이점인 "ALU 2배"는 ALU가 병목이 아닐 때 무의미합니다.

## 확인 방법 (현재 코드 기준)

리포트에서:

- **matmul의 `min DRAM BW`** 가 reuse 스윕 `reuse=0%`의 **DRAM 천장**에 근접하면 → memory-bound 확정.
- **`simple GEMM` vs `shared-memory tiled GEMM`** 두 섹션 비교:
  - tiled 쪽이 더 높고 **INT8 > FP16 차이가 다시 보이면** → 재사용을 늘려 compute-bound로 넘어간 것(정상 회복).
  - tiled도 여전히 ~3으로 같으면 → 타일링이 아직 부족해 메모리에 막힘.

## 해결 방향

GEMM을 **compute-bound로 만들면 INT8 우위가 되살아납니다.** 그것이 shared-memory 타일링의 목적입니다: 로드한 데이터를 workgroup 전체가 재사용하여 로드 수를 급감시켜 연산기를 굶기지 않습니다.

- tiled GEMM의 BEST가 simple보다 오르고 INT8 > FP16 차이가 보이면 → 타일링이 먹힌 것. 타일을 더 키우면 peak(8/16)에 더 근접.
- 그래도 같으면 → 타일을 더 키워(재사용↑) compute-bound로 넘기거나, shared 메모리에서의 `coopMatLoad`가 병목이면 staging 방식을 조정해야 합니다.

## roofline 관점 정리

```
성능(OPS)
  INT8 peak 16 ┤····················  (compute 천장, INT8)
  FP16 peak  8 ┤··········           (compute 천장, FP16)
               │
       ~3 TOPS ┤■■■  FP16 = INT8 (둘 다 memory/load 천장에 걸림)
               │
             0 └───────────────────────────→ 데이터 재사용(arithmetic intensity)
                 MATMUL(저재사용)        WMMA peak(고재사용)
```

- 재사용이 낮은 영역(현재 MATMUL)에서는 **memory/load 천장**이 지배 → 타입 무관하게 동일.
- 재사용이 높은 영역(WMMA peak)에서는 **compute 천장**이 지배 → FP16 8, INT8 16으로 분리.
- 타일링으로 MATMUL의 재사용을 오른쪽으로 밀수록 compute 천장에 가까워지고 타입 차이가 회복됩니다.
