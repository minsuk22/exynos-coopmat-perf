# FMA MATMUL 커널 동작 원리 (`fma_v06.comp`)

이 문서는 WMMA를 쓰지 않는 **FMA(스칼라 곱셈-덧셈) 기반 GEMM** `shaders/fma_v06.comp`가
어떻게 동작하는지, 그리고 **왜 외적(outer product)으로 계산하는지**를 설명합니다.
WMMA 버전은 [wmma-matmul-explained.ko.md](wmma-matmul-explained.ko.md) 참고.

---

## 1. WMMA와의 근본 차이

| | WMMA (`gemm_v07`) | FMA (`fma_v06`) |
|---|------|-----|
| 연산 단위 | 매트릭스 엔진이 16×16을 한 번에 | **스레드가 스칼라 곱셈-덧셈 1개씩** |
| 협력 | subgroup 64개가 한 팀 | **스레드 각자 독립** |
| 출력 담당 | warp가 64×32 | **스레드 1개가 TM×TN** |
| acc 위치 | fragment (lane에 분산) | **레지스터 `acc[TM][TN]`** |

WMMA는 전용 하드웨어(`coopMatMulAdd`)를 호출하지만, FMA는 **평범한 ALU로 직접**
행렬곱을 만듭니다. 그래서 성능의 핵심은 **"한 번 읽은 값을 레지스터에서 최대한
재사용"** 하는 것이고, 그 수단이 **외적(outer product)** 입니다.

---

## 2. 핵심 아이디어 — 레지스터 블로킹 (외적)

**스레드 1개가 TM×TN개 출력을 한꺼번에 맡습니다** (최적 config TM=TN=8 → 64개).

한 스레드가 출력 1개만 맡으면 곱셈 1번 하려고 A·B 값을 매번 읽어야 해서 비효율적입니다.
대신 외적으로:

```
한 스레드가 한 K-스텝(kk)에서:

   A 열 ar[8]        B 행 br[8]
   ┌──┐          ┌──┬──┬──┬──┬──┬──┬──┬──┐
   │a0│          │b0│b1│b2│b3│b4│b5│b6│b7│
   │a1│          └──┴──┴──┴──┴──┴──┴──┴──┘
   │a2│
   │..│   →  acc[i][j] += ar[i] * br[j]   (8×8 = 64번 FMA)
   │a7│
   └──┘
```

**값 16개(ar 8 + br 8)를 읽어 곱셈 64번** → 재사용 = 64/16 = **4.0** (코드 71~79줄).

---

## 3. 왜 외적인가 — 내적과의 비교

행렬곱 `C[i][j] = Σ_k A[i][k]·B[k][j]` 은 루프 순서를 바꿔 두 가지로 계산할 수 있습니다.

### 방식 A — 내적 (inner product, "출력 1개씩 완성")
```
출력 C[i][j] 하나를 잡고 → k를 끝까지 돌며 누적
C[i][j] = A[i][0]·B[0][j] + A[i][1]·B[1][j] + ...
```
→ 곱셈 1번마다 A·B 값 2개 읽음. **재사용 = 0.5**. (우리 `gemm_tiled_scalar`(CUDA 포팅)가
이 방식 — 스레드 1개가 출력 1개.)

### 방식 B — 외적 (outer product, "타일 통째로 조금씩")
```
k를 바깥에 두고 → 한 k에서:
  A 열 ar[0..7], B 행 br[0..7] 읽어서
  acc[i][j] += ar[i]·br[j]   (8×8 = 64개 출력에 동시 누적)
```
→ **값 16개 읽고 곱셈 64번**. (우리 `fma_v06`가 이 방식.)

### 숫자로

| 방식 | 한 스텝 로드 | 곱셈 | 재사용 (곱셈/로드) |
|------|:-----------:|:----:|:------------------:|
| 내적 (1출력) | 2 | 1 | **0.5** |
| 외적 (8×8) | 16 | 64 | **4.0** |

외적은 **`ar[i]`를 TN번(=8) 재사용**(j 전체에), **`br[j]`를 TM번(=8) 재사용**(i 전체에).

재사용 공식:
```
재사용 = (TM × TN) / (TM + TN)
```

### 왜 재사용이 중요한가 (근본 이유)

> GPU의 ALU는 곱셈-덧셈을 매우 빠르게 합니다. 진짜 병목은 **그 곱셈에 먹일 값을
> 공급하는 것**(공유메모리/레지스터 대역폭)입니다.

외적은 **operand를 한 번 레지스터에 올려 타일 전체(TM×TN)에 우려먹는** 구조라 공급
부담을 최소화합니다:

```
한 k 스텝에서 레지스터에 든 것:
  ar[8] + br[8] = 16개   ← 이 16개로
  acc[8][8] = 64개 누산   ← 64번의 FMA를 먹임

다음 k로 가면 ar/br만 새로 16개 교체, acc는 그대로 누적
```

→ acc(64개)는 **레지스터에 계속 머물고**, 매 k마다 **얇은 슬라이스(16개)만 새로 읽어**
곱합니다. "최소 레지스터로 최대 재사용". 내적으로 하면 A 행·B 열을 반복해 읽거나(재사용
없음) 타일 전체를 레지스터에 다 들고 있어야 해서(레지스터 폭발) 비효율적입니다. 외적은
그 사이의 **최적 균형점**입니다.

## 4. 계층 구조

```
출력 2048×2048
  → 128×128 블록 (workgroup, BM×BN)
      → 8×8 thread tile (스레드 1개, TM×TN)
          → 스칼라 FMA 64번 (외적)
```
- threads/wg = (BM/TM)×(BN/TN) = (128/8)×(128/8) = **256 스레드**
- grid = (2048/128)² = **256 workgroups**

## 5. 메인 루프 동작 (코드 57~80줄)

K를 BK=16씩 잘라가며 반복:

### ① 공유메모리로 staging (60~68줄)
```glsl
As[kk][m] = a[(rowBase+m)*K + k0+kk];   // A를 "전치"해서 저장
Bs[kk][n] = b[(k0+kk)*N + colBase+n];   // B는 그대로
```
A를 **전치(transpose)** 해서 올립니다. 이유: 내부 루프에서 한 스레드가 자기 A 열
(`ar[0..7]`)을 읽을 때 **연속된 주소**가 되도록 → 빠른 로드 (74줄).

### ② `barrier()` — 모든 스레드가 staging 끝낼 때까지 대기 (69줄)

### ③ 외적 (71~79줄)
```glsl
for kk in 0..BK:                    // 공유메모리의 K 16개를 하나씩
    ar[i] = As[kk][내 행];           // A 열 8개 → 레지스터
    br[j] = Bs[kk][내 열];           // B 행 8개 → 레지스터
    acc[i][j] += ar[i] * br[j];     // 64번 FMA (외적)
```

### ④ 다음 K-슬랩 (acc는 레지스터에 계속 누적)

64번의 K-슬랩이 끝나면 `acc[8][8]`에 최종 결과가 쌓이고 D에 저장 (82~88줄).

## 6. WMMA vs FMA 한눈 비교

| | WMMA (`gemm_v07`) | FMA (`fma_v06`) |
|---|------|-----|
| 재사용 수단 | 매트릭스 엔진 + 공유메모리 | **레지스터 블로킹(외적)** |
| 한 번에 | `coopMatMulAdd` = 16×16×16 | `ar[i]*br[j]` = 스칼라 1개 |
| 스레드당 출력 | (협력) | TM×TN = 64개 |
| acc 위치 | fragment (분산) | 레지스터 `acc[8][8]` |
| 핵심 파라미터 | C_ROWS×C_COLS | **TM×TN, BK** |

## 7. 벡터화 버전 (`fma_vec4.comp`) — 참고

스칼라 `fma_v06`에 두 가지를 더한 것이 `fma_vec4`입니다:
- **coalesced vec4 전역 로드** (A는 K방향, B는 N방향 연속 → 전치 staging)
- **K를 4개씩 묶어 packed 환산**: `dot(f16vec4)`(packed fp16) / `dotEXT(i8vec4)`(dp4a),
  4 MAC/명령

즉 **타일에 대한 외적은 그대로** 두고, **K 방향 환산만 4-wide 내적(dot)으로** 묶은
하이브리드입니다.

```glsl
for kc in 0..BK/4:                          // K를 4개씩
    av[i] = f16vec4(As[base..base+3]);       // 4개 packed
    bv[j] = f16vec4(Bs[base..base+3]);
    acc[i][j] += dot(av[i], bv[j]);          // 4 MAC / op (packed)
```

---

## 한 줄 정리

> FMA GEMM은 **매트릭스 엔진 없이** 스레드 1개가 **TM×TN(8×8) 출력 타일**을 맡아,
> K를 바깥 루프로 돌리며 한 k마다 **A 열·B 행을 레지스터에 올려 타일 전체를 외적(rank-1)
> 업데이트**합니다. **operand 로드 1번이 여러 FMA를 먹이는(재사용 = TM·TN/(TM+TN) = 4.0)**
> 외적 구조가 성능의 핵심입니다. ALU는 빠르고 operand 공급이 병목이므로, 이 재사용이 곧
> 성능입니다.
