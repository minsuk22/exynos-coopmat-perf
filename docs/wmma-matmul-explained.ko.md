# WMMA MATMUL 커널 동작 원리 (`gemm_v07.comp`)

이 문서는 `MATMUL with WMMA` 테스트가 사용하는 셰이더 `shaders/gemm_v07.comp`가
어떻게 동작하는지 기본 원리부터 차근차근 설명합니다.

---

## 1. GEMM이란 무엇인가

이 커널이 계산하는 건 `D = A × B` (정확히는 `D = α·A·B + β·C`) 입니다. 크기는 모두
1024×1024.

행렬곱의 정의는 단순합니다. **출력 D의 한 원소 `D[i][j]` = A의 i번째 행과 B의 j번째
열의 내적**입니다:

```
D[i][j] = Σ(k=0..1023) A[i][k] × B[k][j]
```

즉 출력 원소 하나당 곱셈 1024번 + 덧셈 1024번. 출력이 1024×1024 ≈ 100만 개이므로
전체 연산량은:

```
2 × M × N × K = 2 × 1024³ ≈ 21억 번 (2.1 GFLOP)
```

여기까지는 그냥 "3중 for문"입니다. 문제는 **이걸 어떻게 빠르게 하느냐**입니다.

---

## 2. 핵심 문제 — "데이터 재사용"

가장 단순하게 짜면 (naive):

```
for i: for j: for k:
    D[i][j] += A[i][k] * B[k][j]   // A, B를 매번 DRAM에서 읽음
```

이게 느린 이유는 **연산보다 메모리 읽기가 병목**이기 때문입니다.

- 곱셈-덧셈 1번 = 매우 빠름 (≈1 사이클)
- 그런데 그걸 하려고 A 원소 1개 + B 원소 1개를 DRAM에서 읽음 = 수백 사이클

DRAM 대역폭은 한정돼 있어서, 데이터를 **한 번 읽어서 최대한 여러 번 재사용**해야
연산기가 놀지 않습니다. 이 비율을 **arithmetic intensity (연산/메모리 비율)** 라고
합니다.

> 핵심 통찰: `A[i][k]` 하나는 D의 **i행 전체**(1024개 출력)를 계산하는 데 쓰입니다.
> 그러니 한 번 읽어서 빠른 메모리(레지스터/공유메모리)에 두고 재사용하면 DRAM 읽기를
> 크게 줄일 수 있습니다.

이 "재사용"을 체계적으로 하는 기법이 **타일링(tiling)** 입니다.

---

## 3. GPU의 실행 계층 (3층 구조)

타일링을 이해하려면 GPU가 일을 나누는 3개 계층을 알아야 합니다:

```
Grid (전체 dispatch)
 └─ Workgroup (작업그룹)              ← 공유메모리(shared/LDS)를 함께 씀
     └─ Subgroup = Warp/Wave (64 스레드)  ← WMMA의 단위!
         └─ Thread (스레드 1개)
```

- **Thread**: 가장 작은 실행 단위
- **Subgroup (= warp, 이 GPU는 wave64라 64 스레드)**: 이 64개 스레드가 한 몸처럼
  협력해서 WMMA 한 번을 실행합니다. 매우 중요합니다.
- **Workgroup**: 여러 subgroup의 묶음. 이 안의 스레드들은 빠른 **공유메모리(LDS)** 를
  함께 쓸 수 있습니다.

---

## 4. WMMA (cooperative matrix)란

`WMMA`는 GPU 안의 **전용 행렬 연산 엔진**(매트릭스 유닛)을 쓰는 명령입니다. 핵심 성질:

> **subgroup 64개 스레드가 협력해서, 16×16×16 짜리 작은 행렬곱을 하드웨어 한 방에
> 처리한다.**

즉 명령 하나(`coopMatMulAdd`)가 이렇게 동작합니다:

```
acc(16×16) += matA(16×16) × matB(16×16)
```

이 16×16짜리 작은 행렬 조각을 **fragment**라고 부릅니다. 사용하는 3개 함수:

- `coopMatLoad`   : 메모리(공유/전역)에서 16×16 fragment를 레지스터로 적재
- `coopMatMulAdd` : `acc += matA × matB` (매트릭스 엔진이 실행)
- `coopMatStore`  : 결과 fragment를 메모리에 저장

GLSL 코드에서 `coopmat<...>` 타입이 바로 이 fragment입니다 (66번 줄의 `acc`,
85·91번 줄의 `matA`/`matB`).

이게 `Peak WMMA` 테스트로 따로 성능을 잰 바로 그 엔진입니다. matmul은 이 16×16 블록을
수천 번 조합해서 1024×1024 전체를 만듭니다.

---

## 5. 이 커널의 3단 타일 계층 (전체 설계)

1024×1024 출력을 3개 계층으로 쪼갭니다:

### ① Block tile — workgroup 1개가 128×128 담당
출력 D(1024×1024)를 **128×128 타일**로 격자 분할합니다. 타일 개수 =
(1024/128)² = **8×8 = 64개**. workgroup 하나가 타일 하나를 책임집니다.

```
TILE_M = WG_H × C_ROWS × lM = 2 × 4 × 16 = 128   (41번 줄)
TILE_N = WG_W × C_COLS × lN = 4 × 2 × 16 = 128   (42번 줄)
```

### ② Warp tile — subgroup 1개가 그 안의 4×2 fragment 담당
한 workgroup에는 subgroup(warp)이 **WG_W×WG_H = 4×2 = 8개** 있습니다. 128×128
타일을 이 8개 warp가 나눠 갖고, **warp 하나는 16×16 fragment를
C_ROWS×C_COLS = 4×2 = 8개** 맡습니다.

확인: 8 warp × 8 fragment × (16×16) = 16384 = 128×128 ✓ (타일을 정확히 덮음)

### ③ WMMA fragment — 16×16×16
warp가 맡은 fragment 하나하나가 `coopMatMulAdd`로 계산되는 16×16 블록입니다.

정리하면:
```
출력 1024×1024
  → 128×128 블록 64개       (workgroup 단위)
      → 16×16 fragment 64개  (warp가 8개씩 분담)
          → WMMA 명령으로 계산
```

---

## 6. 메인 루프의 동작 원리 (재사용이 일어나는 곳)

한 workgroup이 자기 128×128 타일을 계산하는 과정입니다. K(=1024) 방향으로 한 번에 다
못 하니까 **BK=16씩 잘라서** 64번 반복합니다 (71번 줄 `for k0`):

각 K-슬랩마다:

1. **공유메모리로 적재** (74–81번 줄)
   - A의 128×16 슬랩과 B의 16×128 슬랩을 DRAM→LDS(공유메모리)로 복사
   - workgroup의 512 스레드가 협력해서 한 번에 가져옴
2. **`barrier()`** — 모든 스레드가 적재를 끝낼 때까지 대기 (82번 줄)
3. **계산** (84–96번 줄)
   - 각 warp가 **공유메모리에서** 자기 fragment들을 `coopMatLoad`로 읽어
   - `coopMatMulAdd`로 누산기 `acc`에 누적
4. 다음 K-슬랩으로 (acc는 레지스터에 계속 유지)

**여기가 재사용의 핵심입니다:**
> 공유메모리에 올린 A 128×16 슬랩 하나를, 그 workgroup의 **128개 열(전체 TILE_N)** 이
> 전부 재사용합니다. DRAM에서 한 번 읽은 걸 LDS에서 여러 번 우려먹는 거죠. 코드 주석의
> `load:MulAdd = lM/TILE_M + lN/TILE_N = 16/128 + 16/128 = 1/8` 이 그 재사용
> 비율입니다 (DRAM 읽기 1번당 매트릭스 연산 8번).

64번의 K-슬랩이 모두 끝나면 `acc`에 그 타일의 최종 행렬곱 결과가 쌓여 있습니다.

마지막으로 `D = α·acc + β·C`를 계산해서 `coopMatStore`로 DRAM에 씁니다
(100–110번 줄).

---

## 전체를 한 문장으로

> 1024×1024 출력을 **128×128 블록(workgroup)** → **16×16 조각(warp가 8개씩)** 으로
> 쪼개고, K를 16씩 잘라가며 A·B 조각을 **공유메모리에 한 번 올려 여러 번 재사용**하면서,
> 각 조각을 **WMMA 매트릭스 엔진(`coopMatMulAdd`)** 으로 누적 계산하는 커널입니다.

## 주요 파라미터 요약

| 항목 | 값 | 의미 |
|------|----|----|
| M, N, K | 1024 | 행렬 크기 |
| lM, lN, lK | 16 | WMMA fragment 크기 (16×16×16) |
| TILE_M, TILE_N | 128 | workgroup이 담당하는 출력 블록 |
| BK | 16 | 한 번에 공유메모리로 올리는 K 슬랩 |
| C_ROWS × C_COLS | 4 × 2 | warp 하나가 맡는 fragment 수 |
| WG_W × WG_H | 4 × 2 | workgroup 안의 warp 수 (= 8) |
| 스레드/workgroup | 512 | 8 warp × 64 (wave64) |
| workgroup 수 | 64 | (1024/128)² |
| 총 wave 수 | 512 | 64 wg × 8 warp |
| 재사용 비율 | 1/8 | DRAM load : MulAdd |

연산량: `2 × M × N × K = 2 × 1024³ ≈ 2.1 GFLOP` / matmul
